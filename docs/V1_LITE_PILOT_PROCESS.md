# V1.0 Lite 默认试点流程：AI 赋能方案沉淀

> 更新时间：2026-06-29
> 状态：默认试点流程，可被真实业务流程替换。

## 试点目标

把课程笔记、业务背景和用户想法转成一份可评审、可评分、可复用的 AI 赋能方案。

## 第一性原理

AI 赋能平台的最小业务价值不是“搭很多 Agent”，而是让一个复杂想法经过结构化处理、人工审核和质量评价后，变成可落地的工作方案。

## 对抗式审查

- 不把方案生成结果直接当成最终业务决策。
- 不跳过人工审核。
- 不把评分结果当成绝对真理。
- 不要求第一版接入真实飞书通知或生产级部署。

## 输入

```json
{
  "sourceNotes": "课程笔记或业务材料",
  "businessContext": "当前业务目标、部门、角色和约束",
  "desiredOutput": "例如：AI赋能流程方案 / Agent编排建议 / 评分体系设计",
  "riskConcerns": "例如：质量不可控、权限、数据安全、成本"
}
```

## 推荐 Workflow

```text
Start
-> Agent 1：信息抽取与问题建模
-> Agent 2：AI 赋能工作流设计
-> Agent 3：评分与验收体系设计
-> Human Review：业务负责人审核
-> Agent 4：按审核意见修订
-> Evaluation：Rubric 评分
-> End：输出方案文档与验收记录
```

## Agent 草案

### Agent 1：信息抽取与问题建模

- 职责：从课程笔记和业务材料中抽取关键概念、约束、角色、输入输出和风险。
- System Prompt 重点：只做结构化提取，不补不存在的业务事实。
- 产出：问题模型 JSON。

### Agent 2：AI 赋能工作流设计

- 职责：把问题模型转成 Agentic Workflow。
- System Prompt 重点：每个节点必须说明输入、输出、责任边界和是否需要人工审核。
- 产出：流程节点表和节点说明。

### Agent 3：评分与验收体系设计

- 职责：设计 Rubric、权重、门禁和失败处理方式。
- System Prompt 重点：评分必须可观察、可复测，不能只写主观描述。
- 产出：Rubric 草案。

### Agent 4：审核后修订

- 职责：根据人工审核意见修订最终方案。
- System Prompt 重点：保留审核意见与修改理由。
- 产出：最终方案文档。

## Human Review

审核人：业务负责人或产品负责人。

必须检查：

- 业务目标是否清楚。
- Agent 节点是否过多或过少。
- 人工审核节点是否放在高风险位置。
- 评分体系是否能指导真实改进。
- 是否存在数据、安全、权限或成本风险。

## Rubric 草案

| 维度 | 权重 | 评分说明 |
|---|---:|---|
| 业务目标清晰度 | 20% | 是否明确服务哪个业务目标和用户角色 |
| 工作流可执行性 | 25% | 节点输入输出是否清楚，是否能真实流转 |
| 质量评价可操作性 | 25% | Rubric 是否可量化、可复测、可解释 |
| 风险控制 | 20% | 是否覆盖权限、审核、安全、成本和失败恢复 |
| 可迭代性 | 10% | 是否方便后续增加工具、通知和自动化 |

通过门槛：总分 >= 80，且风险控制维度 >= 70。
