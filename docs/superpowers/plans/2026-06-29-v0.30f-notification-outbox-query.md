# V0.30F Notification Outbox Query Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a Workspace-scoped Notification Outbox query API for operations and troubleshooting.

**Architecture:** Add a read-only FastAPI route near the existing Outbox dispatch/requeue routes. It queries `NotificationOutboxRecord` by Workspace, applies bounded filters, and returns existing `NotificationOutboxRead` records without adding database fields.

**Tech Stack:** FastAPI, SQLAlchemy, SQLite JSON payloads, Pydantic, Pytest.

---

## First-Principles Gate

- Bottom-line objective: operators need to find notification facts that require action.
- Minimal objects: Workspace scope, status, channel, error code, bounded result count.
- This API comes before a UI because a page without a stable query source would be a shallow view over missing capability.

## Adversarial Review Gate

- Do not leak cross-Workspace notifications.
- Do not create an unbounded operational query.
- Do not rely on only happy-path sent notifications; failed and pending records matter most.
- Do not add database schema churn for JSON metadata until query needs prove it.
- Do not claim this is an alerting system; it is a query source for one.
