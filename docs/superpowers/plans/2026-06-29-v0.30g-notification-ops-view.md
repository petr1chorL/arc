# V0.30G Notification Ops View Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a frontend Notification Ops view backed by the V0.30F Notification Outbox query API.

**Architecture:** Add a small API module for outbox queries, a dedicated React page for filters and results, and wire it into existing workspace routing and sidebar navigation. Keep the view read-only.

**Tech Stack:** React, React Router, TypeScript, Vitest, Testing Library, FastAPI-backed workspace API.

---

## First-Principles Gate

- Bottom-line objective: operators must see and filter notification facts that require action.
- Minimal interface: status/channel/error-code filters, bounded list, summary counts, error and empty states.
- This page comes after the API because it should expose a real capability, not a static mock.

## Adversarial Review Gate

- Do not imply notifications can be retried or sent from this page yet.
- Do not hide failure evidence; show error text and stable error code.
- Do not make the page a decorative dashboard; it must be dense and operational.
- Do not bypass Workspace routing or hard-code Workspace IDs.
- Do not add a second source of truth for status counts; derive them from loaded records.
