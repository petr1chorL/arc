# V0.31A Notification Channel Assets Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a Workspace-scoped notification channel asset API so real external notification adapters can later be governed by Workspace configuration instead of hard-coded adapter registration.

**Architecture:** Add a small `notification_channels` table, Pydantic schemas, Workspace API routes, and focused backend tests. Keep the existing V0.30 dispatch behavior unchanged and do not connect to external networks.

**Tech Stack:** FastAPI, SQLAlchemy, Pydantic, Pytest, SQLite-compatible migrations.

---

## First-Principles Gate

- Bottom-line objective: before real external notification delivery, the platform must know which channels a Workspace has configured and whether each is enabled.
- Minimal objects: channel asset id, workspace id, name, channel type, status, config object, secretRef label, timestamps.
- Minimal closed loop: create channel -> list channel -> disable channel with audit evidence.
- This is a governance boundary, not a delivery adapter.

## Adversarial Review Gate

- Do not save or expose secret values.
- Do not call this real Feishu/email/Webhook delivery.
- Do not bypass Workspace scope.
- Do not allow duplicate names inside one Workspace.
- Do not wire this into dispatch routing until a later version can verify adapter semantics.
