# V0.30H Notification Requeue View Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let operators requeue a failed Notification Outbox record from the notification operations page with an explicit audit reason.

**Architecture:** Reuse the existing V0.30C backend requeue API. Add a frontend API wrapper, then add a small inline action state to the V0.30G notification list. Keep the action available only for failed records and refresh the current filtered list after success.

**Tech Stack:** React, React Router, TypeScript, Vitest, Testing Library, FastAPI-backed workspace API.

---

## First-Principles Gate

- Bottom-line objective: convert a verified failed notification into a pending notification without leaving the Workspace operations context.
- Minimal required flow: failed notification -> reason capture -> requeue API -> refreshed list.
- Reason capture is not decoration; it is the minimum durable explanation for an auditable recovery action.

## Adversarial Review Gate

- Do not expose requeue on `sent` or `pending` notifications.
- Do not submit empty reasons.
- Do not hide API errors or clear the reason when the action fails.
- Do not claim this is automatic retry, batch recovery, or real external delivery.
- Do not create a duplicate endpoint or bypass the Workspace-scoped API.
