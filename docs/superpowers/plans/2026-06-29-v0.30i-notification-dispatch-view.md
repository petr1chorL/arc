# V0.30I Notification Dispatch View Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let operators trigger one Notification Outbox dispatch pass from the notification operations page and see the processing summary.

**Architecture:** Reuse the existing V0.30A backend dispatch endpoint. Add typed frontend API support, add a compact dispatch action to the Notifications page, show the latest dispatch summary, and refresh the current filtered list after success.

**Tech Stack:** React, React Router, TypeScript, Vitest, Testing Library, FastAPI-backed workspace API.

---

## First-Principles Gate

- Bottom-line objective: pending notifications should be handable to the existing dispatch service from the operations page.
- Minimal required flow: trigger dispatch -> show processed/sent/failed -> refresh visible Outbox records.
- This is an operator trigger for the existing service, not a promise of real external delivery.

## Adversarial Review Gate

- Do not label the action as guaranteed external send.
- Do not dispatch individual records from the frontend; call the Workspace dispatch endpoint once.
- Do not show stale success summary when the dispatch API fails.
- Do not add a second notification sending model in the frontend.
- Do not add automatic polling or hidden repeated sends.
