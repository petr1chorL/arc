# V0.30E Notification Dispatch Metadata Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add structured channel and failure-code metadata to Notification Outbox dispatch results.

**Architecture:** Extend the existing dispatch result dataclass and schema without adding database fields. The dispatch service writes the metadata into `payload.dispatch`, while API responses expose the same values for immediate operational feedback.

**Tech Stack:** FastAPI, Pydantic, SQLAlchemy JSON payloads, Pytest.

---

## First-Principles Gate

- Bottom-line objective: failures must be classifiable without parsing free-text errors.
- Minimal objects: channel, stable error code, human-readable error, and persisted dispatch metadata.
- This is more fundamental than dashboards or retries because those features need stable failure facts first.

## Adversarial Review Gate

- Do not replace human-readable `error`; add stable `errorCode` alongside it.
- Do not add columns before the metadata proves useful.
- Do not break existing dict-based fake dispatchers.
- Do not claim this implements retry decisions; it only supplies the facts retries would need.
