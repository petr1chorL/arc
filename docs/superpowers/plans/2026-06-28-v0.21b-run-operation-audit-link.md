# V0.21B Run 操作历史跳转审计日志实施计划

## 目标

让用户在运行中心查看 Run 操作历史时，可以从单条历史记录直接跳转到同一 Trace ID 的审计日志，减少排查运行操作来源时的页面切换成本。

## 文件范围

- `apps/api/app/schemas.py`
- `apps/api/app/main.py`
- `apps/api/tests/test_execution_api.py`
- `src/types.ts`
- `src/pages/Runs.tsx`
- `src/pages/Runs.test.tsx`
- `src/index.css`
- `docs/ACCEPTANCE_V0.21B.md`
- `docs/CURRENT_IMPLEMENTATION.md`

## 实施步骤

1. 后端 RED：在 Run 操作历史测试中断言返回 `traceId`。
2. 后端 GREEN：`RunOperationHistoryEventRead` 增加 `traceId`，并从 `AuditEventRecord.trace_id` 填充。
3. 前端 RED：在 Runs 页面测试中断言操作历史存在“查看审计”链接。
4. 前端 GREEN：类型补齐 `traceId`，操作历史条目渲染审计链接。
5. 文档：补 V0.21B 验收文档与当前实现说明。
6. 验证：focused 测试、完整测试、lint、build、浏览器验收。

## 验收标准

- 操作历史接口返回的每条记录包含 `traceId`。
- Runs 页面操作历史条目有“查看审计”链接。
- 链接携带当前 Workspace slug 和 URL 编码后的 Trace ID。
- 点击后进入审计日志 Trace 过滤页面。
- 现有 Run 详情、重跑、恢复、批量操作测试不回归。
