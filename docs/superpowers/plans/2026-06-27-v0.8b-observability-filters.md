# V0.8B Observability Filters Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add shareable observability filters so operators can narrow runs by status, workflow name, and risk level without losing the selected run context.

**Architecture:** Keep the existing observability APIs unchanged. Load the workspace overview once, filter `recentRuns` and `risks` in `src/pages/Observability.tsx`, synchronize filter state to URL query parameters, and preserve the current run detail fetch behavior.

**Tech Stack:** React, React Router search params, Vitest, Testing Library, TypeScript, existing FastAPI observability APIs.

### Task 2: Implement Filters

**Files:**
- Modify: `src/pages/Observability.tsx`

- [x] **Step 1: Add search param state**

Use `useSearchParams` from React Router. Supported query keys:

```text
status
workflow
risk
runId
```

Risk values:

```text
all
critical
warning
```

- [x] **Step 2: Filter runs and risks**

Filter `overview.recentRuns` by:

```text
status === 全部 or displayStatus(run.status) === status
workflow text included in workflowName
risk === all or run.priority === risk
```

Filter `overview.risks` by matching the filtered run IDs and risk severity.

- [x] **Step 3: Preserve selected run**

When filtered data changes, choose:

```text
runId query if it exists in filtered runs
current selected run if still visible
first filtered risk run
first filtered recent run
empty string
```

When clicking a run or risk card, update `runId` in the URL.
