# V0.24F Run Input Schema Validation Implementation Plan

## Goal

Validate workflow run input on the backend against the published workflow version's frozen `inputSchema`, so schema-backed workflows cannot be bypassed by direct API calls.

## Scope

- [x] Add focused backend tests for missing required fields, type mismatch, and valid input.
- [x] Resolve the requested workflow version before execution.
- [x] Validate only simple one-level object schemas.
- [x] Return `422` before creating a Run or calling the model when validation fails.
- [x] Keep empty or unsupported schemas compatible with existing free-text behavior.
- [x] Document acceptance evidence.

## Design

- Use `WorkflowVersionRecord.workflow_snapshot["inputSchema"]` as the source of truth.
- Parse `RunCreate.input` as JSON only when the schema is a simple object with supported properties or required fields.
- Validate:
  - JSON must parse to an object.
  - Required fields must exist and non-empty strings are required for string fields.
  - `string`, `number`, `integer`, `boolean` field types must match JSON values.
- Keep additional fields allowed.
- Skip validation for unsupported property types so legacy workflows are not blocked.

## Out of Scope

- Full JSON Schema engine.
- Nested objects, arrays, enum, format, pattern, defaults, min/max and additionalProperties.
- Frontend changes.
- Data Object assets.
