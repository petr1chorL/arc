# V0.31C Notification Channel Dispatch Guard Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make default Notification Outbox dispatch explain Workspace channel asset state before delegating to code adapters.

**Architecture:** Add optional channel asset preflight to `NotificationOutboxDispatchService`. Enable it only for the default app dispatcher path, leaving explicitly injected test/custom dispatchers unchanged. Persist preflight failures through the existing dispatch result payload.

**Tech Stack:** FastAPI, SQLAlchemy, Pytest.

---

## First-Principles Gate

- Bottom-line objective: failed dispatch results must identify governance causes.
- Minimal distinctions: channel asset missing, channel asset disabled, adapter missing.
- `in_app` remains an internal default channel.

## Adversarial Review Gate

- Do not claim active assets can send real external messages.
- Do not parse or read secret values.
- Do not require channel assets for explicit injected adapter tests.
- Do not add schema fields when existing dispatch metadata can carry the evidence.
