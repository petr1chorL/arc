# V0.25A Data Object Definitions Implementation Plan

## Goal

Introduce Workspace-level Data Object Definition and immutable Data Object Version backend assets.

## Scope

- [x] Add SQLAlchemy models for Data Object Definition and Data Object Version.
- [x] Add Pydantic create/read/version schemas.
- [x] Add migration coverage for existing databases.
- [x] Add API endpoints:
  - `GET /data-objects`
  - `POST /data-objects`
  - `POST /data-objects/{definitionId}/publish`
- [x] Add tests for create/list, duplicate name, workspace isolation, and immutable version snapshot.
- [x] Document acceptance evidence.

## Design

- Definition stores `name`, `description`, `schema`, `status`, `version`, `createdBy`, timestamps and `workspaceId`.
- Version stores `definitionId`, `version`, immutable `snapshot` and `workspaceId`.
- Version numbering reuses `next_version(count)`.
- Schema must be a JSON object, but full JSON Schema validation is out of scope.
- Use existing Workspace authorization style: read uses `asset.read`, write/publish uses `agent.write`.

## Out of Scope

- Frontend page.
- Node binding.
- Runtime Artifact instances.
- Data Object impact analysis.
- Deactivation and migration tools.
