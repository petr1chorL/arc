# V0.7A 身份、Workspace 与访问控制设计

> 状态：已批准  
> 日期：2026-06-25

## 1. 背景

ARC.ONE V0.6 已具备 Agent、工作流、运行和人工协作的单机闭环，但所有 API
默认匿名可访问，页面中的 Workspace 与生产环境只是展示文本，Reviewer 也是
独立本地目录项。任何访问者都可以查看或修改全部资源，审核动作依赖前端选择的
“当前操作者”，无法形成可信身份、权限边界和完整审计。

V0.7 拆为两个可独立验收版本：

- V0.7A：身份、组织、Workspace、成员、RBAC、Reviewer 用户绑定和平台审计。
- V0.7B：密钥托管、环境隔离、预算配额、模型与工具白名单和安全策略。

本文只设计 V0.7A。

## 2. 目标

V0.7A 必须实现：

1. 用户使用邮箱和密码登录，服务端建立可撤销 Session。
2. 平台采用单组织、多 Workspace、多成员模型。
3. 所有业务资产和运行数据归属明确的 Workspace。
4. 用户通过 Workspace 成员关系获得一个固定平台角色。
5. 后端根据 capability 强制执行权限，默认拒绝。
6. Reviewer 成为绑定真实用户的额外业务资格。
7. 关键成功、失败和权限拒绝形成不可修改的审计事件。
8. 现有数据和历史审核关系可迁移，不丢失稳定标识。
9. 登录、Workspace 切换、成员管理和审计查询具有完整前端路径。
10. 为后续 OIDC、飞书 SSO 和 Keycloak 接入保留认证端口。

## 3. 非目标

V0.7A 不包含：

- 多组织注册、跨企业租户开通和计费。
- 飞书、Keycloak 或其他 OIDC Provider 的真实联调。
- 自定义角色和可视化权限编辑器。
- 跨 Workspace 共享、移动或复制资产。
- 成员彻底删除、批量导入和企业组织架构同步。
- API Key、模型密钥和工具凭证托管。
- 开发、测试、生产环境隔离。
- 预算、配额、模型白名单和 OPA 策略。
- Kubernetes、高可用 Session 存储和多节点部署。

V0.7A 是本地单体架构上的身份与权限闭环，不应描述为企业 SSO 或生产级
多租户认证系统。

## 4. 已确认原则

### 4.1 身份、授权和审核资格分离

- `User` 回答“谁在操作”。
- `WorkspaceMembership` 回答“用户在哪个 Workspace 具有什么平台角色”。
- `Reviewer` 回答“用户是否有资格处理人工任务”。
- `AuthorizationService` 回答“当前用户能否执行某项 capability”。

Reviewer 不是账号，也不是平台角色。任意平台角色只有获得审核资格后，才能处理
Human Task。

### 4.2 Workspace 显式进入资源路径

Web 路由使用：

```text
/w/{workspaceSlug}/agents
/w/{workspaceSlug}/workflows
/w/{workspaceSlug}/runs
/w/{workspaceSlug}/reviews
/w/{workspaceSlug}/settings/members
/w/{workspaceSlug}/settings/audit
```

API 使用：

```text
/api/workspaces/{workspaceId}/agents
/api/workspaces/{workspaceId}/workflows
/api/workspaces/{workspaceId}/runs
/api/workspaces/{workspaceId}/human-tasks
```

当前 Workspace 不存入隐藏 Cookie。URL、请求、审计和资源归属必须保持一致。

### 4.3 API 是安全边界

前端根据权限隐藏命令或进入只读状态，只用于减少误操作。每个受保护 API 必须：

1. 验证 Session。
2. 解析当前 User。
3. 验证 Organization 和 Workspace。
4. 解析 WorkspaceMembership。
5. 校验 capability 或 Reviewer 资格。
6. 限定数据库查询的 `workspace_id`。
7. 在关键动作后写入审计事件。

## 5. 总体架构

### 5.1 AuthenticationService

负责：

- 邮箱密码登录。
- 密码哈希验证与更新。
- Session 创建、续期、撤销和过期。
- 登录失败计数与临时锁定。
- 邀请令牌创建、重发、撤销和激活。
- 密码修改后撤销该用户全部 Session。

密码使用 Argon2id。Session 与邀请使用密码学安全随机令牌，数据库只保存令牌摘要，
不保存可直接使用的明文令牌。

### 5.2 RequestContext

每个受保护请求解析出：

```text
current_user
organization
workspace
workspace_membership
session
```

组织管理员在组织内拥有所有 Workspace 管理权限。普通用户必须具有有效的
WorkspaceMembership。

### 5.3 AuthorizationService

AuthorizationService 是 capability 判断的唯一入口。业务路由不能自行比较角色字符串。

接口语义：

```text
require_capability(context, capability)
require_reviewer(context, workspace_id)
require_expert_reviewer(context, workspace_id)
```

拒绝时：

- 未登录返回 `401`。
- 已登录但缺少普通能力返回 `403`。
- 资源不存在或跨 Workspace 探测返回 `404`。
- 权限拒绝写入 AuditEvent。

### 5.4 WorkspaceService

负责：

- 创建和查询 Workspace。
- 生成并校验唯一 slug。
- 查询当前用户可访问的 Workspace。
- 记录和恢复用户最近访问的 Workspace。
- 保护组织和 Workspace 管理边界。

### 5.5 MembershipService

负责：

- 邀请成员并指定角色。
- 生成可由管理员复制的一次性激活链接。
- 重发和撤销未激活邀请。
- 激活成员。
- 修改成员角色。
- 启用和停用成员。
- 授予或撤销 Reviewer、专家资格。
- 禁止管理员停用自己。
- 保证每个 Workspace 至少保留一名有效 Workspace 管理员。

### 5.6 AuditService

AuditService 记录认证、成员、Workspace、资产、运行和审核关键事件。

对于受保护的业务写操作，业务变更与审计事件必须在同一数据库事务中提交；
审计写入失败时，业务变更也失败。登录失败等没有业务事务的安全事件单独提交。

平台不提供修改或删除审计事件的 API。

### 5.7 AuthProviderPort

认证入口通过端口抽象：

```text
LocalPasswordAuthProvider
FutureOidcAuthProvider
```

V0.7A 只实现 LocalPasswordAuthProvider。OIDC 端口只定义身份归一化边界，不创建
未使用的 Provider 配置页面和外部身份表。

## 6. 领域与持久化模型

### 6.1 OrganizationRecord

字段：

- `id`
- `name`
- `slug`
- `status`
- `created_at`
- `updated_at`

V0.7A 只有一个有效组织，但所有 Workspace 和 User 均显式记录组织归属。

### 6.2 UserRecord

字段：

- `id`
- `organization_id`
- `email`，迁移待补录用户可为空
- `normalized_email`
- `display_name`
- `password_hash`
- `status`：`pending_email`、`invited`、`active`、`disabled`
- `is_organization_admin`
- `failed_login_count`
- `locked_until`
- `password_changed_at`
- `last_login_at`
- `last_workspace_id`
- `created_at`
- `updated_at`

有效邮箱在组织内唯一。所有登录比较使用规范化邮箱。

### 6.3 WorkspaceRecord

字段：

- `id`
- `organization_id`
- `name`
- `slug`
- `status`
- `created_by`
- `created_at`
- `updated_at`

`organization_id + slug` 唯一。

### 6.4 WorkspaceMembershipRecord

字段：

- `id`
- `workspace_id`
- `user_id`
- `role`
- `status`：`invited`、`active`、`disabled`
- `invited_by`
- `activated_at`
- `created_at`
- `updated_at`

`workspace_id + user_id` 唯一。每个成员关系只有一个 Workspace 角色。

### 6.5 SessionRecord

字段：

- `id`
- `user_id`
- `token_digest`
- `csrf_digest`
- `created_at`
- `last_seen_at`
- `idle_expires_at`
- `absolute_expires_at`
- `revoked_at`
- `revoked_reason`
- `ip_address`
- `user_agent`

Session Cookie 保存不透明随机令牌，数据库只保存摘要。

### 6.6 InvitationRecord

字段：

- `id`
- `organization_id`
- `workspace_id`
- `user_id`
- `role`
- `token_digest`
- `expires_at`
- `used_at`
- `revoked_at`
- `created_by`
- `created_at`

同一邀请令牌只能成功激活一次。

### 6.7 ReviewerRecord

保留现有 Reviewer ID 和历史关系，新增：

- `user_id`
- `workspace_id`

Reviewer 必须绑定 User 后才能产生新的审核动作。迁移生成的 `pending_email` User
允许显示历史，但不能认领、转交、决定或确认 Golden Sample。

### 6.8 Workspace 归属

以下记录增加 `workspace_id`：

- Agent 和 AgentVersion。
- Workflow 和 WorkflowVersion。
- WorkflowRun 和 NodeRun。
- Artifact、ArtifactVersion 和 ArtifactDiff。
- Reviewer、ReviewGroup 和 ReviewGroupMember。
- HumanTask、ReviewDecision 和 ResumeRequest。
- NotificationOutbox。
- FeedbackCandidate 和 GoldenSample。
- AuditEvent。

创建子记录时继承聚合根的 Workspace，不接受客户端自行指定冲突的 Workspace。

### 6.9 AuditEventRecord

现有仅服务 Human Task 的审计表扩展为平台审计事件，保留旧字段兼容历史记录，
并新增：

- `organization_id`
- `workspace_id`
- `actor_user_id`
- `session_id`
- `action`
- `target_type`
- `target_id`
- `outcome`：`success`、`failure`、`denied`
- `request_id`
- `ip_address`
- `metadata`

旧 Human Task 事件继续保留 `human_task_id`、前后状态和原因。

## 7. 角色与 Capability

### 7.1 固定角色

Workspace 角色按权限逐级包含：

```text
viewer < operator < builder < workspace_admin
```

组织管理员不是 WorkspaceMembership 的第五种角色，而是 User 的组织级管理员标记；
其在组织内自动拥有所有 Workspace capability。

### 7.2 权限矩阵

| Capability | 观察者 | 运行者 | 构建者 | Workspace 管理员 |
|---|---:|---:|---:|---:|
| `asset.read` | ✓ | ✓ | ✓ | ✓ |
| `run.read` | ✓ | ✓ | ✓ | ✓ |
| `audit.read` |  |  |  | ✓ |
| `run.execute` |  | ✓ | ✓ | ✓ |
| `agent.write` |  |  | ✓ | ✓ |
| `agent.publish` |  |  | ✓ | ✓ |
| `workflow.write` |  |  | ✓ | ✓ |
| `workflow.publish` |  |  | ✓ | ✓ |
| `asset.deactivate` |  |  |  | ✓ |
| `member.manage` |  |  |  | ✓ |
| `reviewer.manage` |  |  |  | ✓ |
| `workspace.manage` |  |  |  | ✓ |
| `audit.export` |  |  |  | ✓ |

组织管理员额外拥有：

- 创建 Workspace。
- 管理组织级成员状态。
- 管理任意 Workspace 的成员。
- 查看和导出组织内审计事件。

### 7.3 审核能力

以下能力不由平台角色自动授予：

- `review.act`：要求有效 Reviewer。
- `golden_sample.confirm`：要求有效且具备专家资格的 Reviewer。

Reviewer 仍必须是目标 Workspace 的有效成员。

## 8. 认证与会话规则

### 8.1 登录

- 使用邮箱和密码。
- 登录失败返回统一错误，不区分邮箱不存在、密码错误或账号停用。
- 连续 5 次失败锁定 15 分钟。
- 成功登录清零失败次数并写入 AuditEvent。
- 停用、未激活和锁定用户不能建立 Session。

### 8.2 Session 生命周期

- 连续 8 小时无活动后过期。
- 有效请求滑动续期 8 小时。
- Session 创建后绝对最长 7 天。
- 修改密码或停用用户立即撤销该用户全部 Session。
- 退出只撤销当前 Session。

### 8.3 Cookie 与 CSRF

Session Cookie：

- `HttpOnly`
- `SameSite=Lax`
- `Path=/`
- 生产 HTTPS 环境强制 `Secure`

登录成功时同时设置非 `HttpOnly` 的 `arc_one_csrf` Cookie，使用 `SameSite=Lax`、
`Path=/`，并在生产 HTTPS 环境强制 `Secure`。前端从该 Cookie 读取值并通过
`X-CSRF-Token` 请求头提交；服务端与 Session 中保存的摘要匹配。Session Cookie
仍保持 `HttpOnly`，前端无法读取。登录、邀请预览和邀请激活使用独立的 Origin
检查与请求限流。

### 8.4 邀请

- 邀请有效期 72 小时。
- 令牌只能使用一次。
- 重发邀请撤销旧令牌并创建新令牌。
- 撤销后不能激活。
- 激活时设置密码和显示名称，并创建有效 Membership。
- 正式迁移不生成默认密码。
- V0.7A 不发送邮件或外部消息。成员页面只向有权限的管理员展示一次性激活链接，
  由管理员通过现有企业渠道发送。
- 激活链接中的明文令牌只在创建或重发成功响应中返回一次，后续查询只显示邀请状态。

## 10. 前端设计

### 10.1 路由

新增公共页面：

- `/login`
- `/activate/:token`

新增受保护路由：

- `/w/:workspaceSlug/*`

访问 `/` 时：

1. 未登录跳转 `/login`。
2. 已登录进入最近访问且仍有权限的 Workspace。
3. 最近 Workspace 不可用时进入第一个有效 Workspace。
4. 没有任何 Membership 时显示无可访问 Workspace 状态。

### 10.2 AuthProvider

前端 AuthProvider 只维护：

- 当前 User。
- Session 状态。
- 可访问 Workspace 列表。
- 当前 Membership 与 capabilities。

API 客户端统一读取 `arc_one_csrf` Cookie 并发送 CSRF Header。页面不得自行拼装
认证头或读取 Session Cookie。

### 10.3 ProtectedRoute 与 CapabilityGuard

- `ProtectedRoute` 处理未登录、Session 加载和失效跳转。
- `WorkspaceRoute` 校验 slug 是否属于当前用户。
- `CapabilityGuard` 控制命令可见性和只读状态。

CapabilityGuard 不能替代 API 校验。

### 10.4 应用外壳

侧栏底部 Workspace 切换器展示用户可访问空间。顶部栏增加用户菜单，包含：

- 当前用户名称和邮箱。
- 当前 Workspace 角色。
- Session 到期提示。
- 退出。

Workspace 管理员和组织管理员看到：

- 成员与权限。
- 审计日志。

### 10.5 成员与权限页面

表格展示：

- 姓名与邮箱。
- 平台角色。
- Reviewer 与专家资格。
- 激活状态。
- 最近登录时间。
- 可用操作。

支持邀请、复制激活链接、重发、撤销、改角色、启停和审核资格管理。最后一名管理员、
自身停用等受保护操作显示明确原因。

### 10.6 审计日志页面

使用高密度表格展示：

- 时间。
- 操作者。
- 动作。
- 对象。
- Workspace。
- 结果。
- 请求 ID。

支持筛选、详情展开和 CSV 导出，不提供编辑或删除。

## 11. 数据迁移

迁移按以下顺序执行：

1. 创建 Organization、User、Workspace、Membership、Session 和 Invitation 表。
2. 创建默认组织“安克创新”和默认 Workspace“AI 能力中心”。
3. 为现有业务表增加可空 `workspace_id`。
4. 将现有 Agent、工作流、运行、产出物、审核和评估数据回填到默认 Workspace。
5. 为现有 Reviewer 创建 `pending_email` User，并保存 `Reviewer.user_id`。
6. 为 ReviewGroup 和历史审核子记录回填 Workspace。
7. 验证所有现有聚合根都有 Workspace 后，应用层拒绝创建无 Workspace 新记录。
8. 通过初始化命令创建首位组织管理员，不在代码或数据库中写入默认密码。

当前仓库仍使用 SQLite 轻量增量迁移。V0.7A 扩展幂等迁移函数并保留备份说明；
PostgreSQL 与 Alembic 正式迁移在生产部署阶段完成，不能把本轮迁移描述为跨数据库
生产迁移。

## 12. 异常处理

| 场景 | HTTP | 对外行为 |
|---|---:|---|
| 未登录或 Session 失效 | 401 | 前端清理身份并跳转登录 |
| CSRF 缺失或错误 | 403 | 显示会话安全校验失败 |
| 已登录但无 capability | 403 | 显示无权限，并记录 denied 审计 |
| 资源不存在或跨 Workspace | 404 | 不泄露其他 Workspace 资源 |
| 邀请已使用、撤销或过期 | 409 | 显示邀请不可用 |
| 最后一名管理员被降级或停用 | 409 | 返回明确保护原因 |
| 登录锁定 | 429 | 返回稍后重试，不暴露账号是否存在 |
| 字段校验失败 | 422 | 返回字段级错误 |

Session 在用户编辑表单期间失效时，前端保留未提交内容，提示重新登录；重新认证后
由用户主动再次提交，不自动重放写请求。

## 13. 审计范围

必须记录：

- 登录成功、失败、锁定、退出和 Session 失效。
- 邀请创建、激活链接复制、重发、撤销和激活。
- 成员启停、角色变化、Reviewer 与专家资格变化。
- Workspace 创建和设置变化。
- Agent、工作流的创建、编辑、发布和停用。
- 工作流运行、终止和重试。
- Human Task 认领、转交和审核决定。
- Golden Sample 确认。
- 权限拒绝。
- 审计导出。

敏感字段不得进入 metadata：

- 明文密码。
- Session 和邀请令牌。
- CSRF Token。
- 模型 API Key。

## 14. 测试设计

### 14.1 后端聚焦测试

- 密码哈希不等于明文，正确与错误密码行为明确。
- 登录失败计数、15 分钟锁定与成功重置。
- Session 8 小时滑动过期、7 天绝对过期和主动撤销。
- 修改密码与停用用户撤销全部 Session。
- CSRF 缺失、错误和正确请求。
- 邀请过期、撤销、重发、重复激活。
- 明文邀请令牌只在创建或重发响应中出现一次。
- 每个 Workspace 至少保留一名管理员。
- 组织管理员和四种 Workspace 角色的 capability 矩阵。
- 跨 Workspace 查询和修改返回 404。
- Reviewer 与专家资格校验。
- 关键成功、失败和 denied 事件写入审计。
- 审计事件没有更新和删除入口。

### 14.2 迁移测试

- 旧 V0.6 SQLite 数据库可升级。
- 默认组织和 Workspace 只创建一次。
- 现有资产全部归入默认 Workspace。
- 现有 Reviewer ID 和历史 ReviewDecision 关系保持不变。
- 重复运行迁移不会创建重复数据。

### 14.3 前端测试

- 未登录访问受保护路由跳转登录。
- 登录成功进入最近 Workspace。
- Workspace 切换更新 URL 和请求路径。
- 权限不足时按钮隐藏或表单只读。
- Session 失效保留未提交表单并提示重新登录。
- 成员邀请、角色修改、启停和 Reviewer 资格错误状态。
- 激活链接复制后显示明确反馈，刷新页面后不再返回明文令牌。
- 审计筛选与导出入口。

### 14.4 浏览器验收

完整路径：

```text
创建组织管理员
→ 登录
→ 创建 Workspace
→ 邀请构建者
→ 复制激活链接并通过现有企业渠道发送
→ 激活账号
→ 构建者创建并发布 Agent
→ 运行者运行工作流但不能编辑
→ Reviewer 处理 Human Task
→ 观察者只读
→ 管理员查询审计日志
→ 用户退出后受保护页面不可访问
```

另行验证跨 Workspace 资源访问、最后管理员保护、停用用户旧 Session 和移动端布局。

## 15. 实施切片

V0.7A 按端到端价值拆成：

1. Bootstrap 管理员、登录、Session、CSRF 和受保护路由。
2. 默认组织、Workspace、Membership 与现有资产迁移。
3. 固定角色、capability 授权与业务 API Workspace 化。
4. 成员邀请、激活、角色和启停管理。
5. Reviewer 用户绑定与 Human Task 真实操作者。
6. 平台审计查询、筛选和导出。
7. 跨角色、跨 Workspace 和浏览器完整验收。

每个切片必须先形成失败测试，再实现最小行为，并在完成前运行后端全量测试、
前端全量测试、Lint、Build 和浏览器验收。

## 16. 完成标准

V0.7A 只有满足以下条件才完成：

- 匿名用户无法访问业务 API。
- 用户只能看到有 Membership 的 Workspace。
- 所有业务查询和写入受 `workspace_id` 限制。
- 权限矩阵在 API 层完整执行。
- Reviewer 操作来自当前登录用户，前端不能伪造操作者。
- Session、CSRF、锁定和邀请边界测试通过。
- 现有 V0.6 数据迁移后可继续读取。
- 关键动作具有不可修改审计事件。
- 登录、切换、邀请、激活、授权、审核和退出浏览器路径通过。
- `docs/CURRENT_IMPLEMENTATION.md` 准确描述能力和剩余限制。
- 不把 V0.7A 描述为已完成企业 SSO、多租户 SaaS 或生产高可用认证。
