# V0.8A 运行观测与异常排障设计

## 目标

V0.8A 建设轻量运行观测与异常排障中心，让平台管理员和业务运营可以从一个入口判断：

- 当前 Workspace 有多少运行成功、失败、需介入或恢复失败。
- 哪些运行最需要处理。
- 某次运行卡在哪个节点，耗时、错误、Token 和成本是什么。
- 等待人工审核的运行应该跳转到哪里处理。

## 范围

本阶段只做 V0.8A 第一刀：运行排障中心。

包含：

- Workspace 级观测概览 API。
- 单次运行排障详情 API。
- 前端观测中心页面。
- 左侧导航入口。
- 状态筛选和工作流名称搜索。
- 异常优先展示和空状态引导。

不包含：

- OpenTelemetry、Prometheus、Grafana、Loki、Tempo、Langfuse。
- WebSocket/SSE 实时推送。
- 飞书、邮件或 Webhook 通知。
- 完整 Trace Span 模型。
- 成本预算审批。
- V0.9 真实业务试点效果验证。

## 架构决策

### 1. 轻量数据库聚合

V0.8A 直接读取现有 SQLAlchemy 记录：

- `WorkflowRunRecord`
- `NodeRunRecord`
- `ArtifactRecord`
- `HumanTaskRecord`
- `AuditEventRecord`
- `NotificationOutboxRecord`

这样可以先验证“观测页面是否有用”，不提前引入外部观测基础设施。

### 2. Workspace 路径和权限

所有新 API 放在 Workspace 路径下：

```text
GET /api/workspaces/{workspace_id}/observability/overview
GET /api/workspaces/{workspace_id}/observability/runs/{run_id}
```

权限使用现有 `asset.read` 能力。前端只减少误操作，后端仍是安全边界。

### 3. 异常优先

概览 API 不只是返回全部运行列表，而是显式计算优先级：

1. 失败。
2. 恢复失败。
3. 需介入。
4. 运行中或等待中。
5. 已完成。

前端默认按这个优先级展示。

### 4. 处理建议

后端返回面向用户的 `nextAction`：

- 失败：进入运行详情查看失败节点和错误。
- 需介入：进入人工审核页处理 Human Task。
- 恢复失败：进入人工审核页重试恢复或重新提交决定。
- 已完成：查看产出物和耗时。

建议只基于已有状态生成，不引入规则引擎。

## API 草案

### ObservabilityOverview

```ts
interface ObservabilityOverview {
  totals: {
    runs: number
    succeeded: number
    failed: number
    waitingForHuman: number
    resumeFailed: number
    averageDurationMs: number | null
    totalPromptTokens: number
    totalCompletionTokens: number
    totalCostUsd: number
  }
  risks: ObservabilityRisk[]
  recentRuns: ObservabilityRunSummary[]
}
```

### ObservabilityRunSummary

```ts
interface ObservabilityRunSummary {
  id: string
  workflowName: string
  status: string
  currentNode: string
  startedAt: string
  completedAt: string | null
  durationMs: number | null
  score: number | null
  costUsd: number
  promptTokens: number
  completionTokens: number
  priority: 'critical' | 'warning' | 'normal'
  nextAction: string
}
```

### ObservabilityRunDetail

```ts
interface ObservabilityRunDetail extends ObservabilityRunSummary {
  nodes: ObservabilityNodeRun[]
  humanTasks: ObservabilityHumanTask[]
  auditEvents: ObservabilityAuditEvent[]
}
```

## 前端设计

页面名称：`Observability`

路由：

```text
/observability
```

Workspace 路径：

```text
/w/:workspaceSlug/observability
```

页面结构：

- 顶部指标条：运行总数、失败、需介入、恢复失败、平均耗时、Token/成本。
- 左侧或上方筛选：状态、工作流名称。
- 中间运行列表：异常优先。
- 右侧详情：节点时间线、错误、Token、成本、处理建议。
- 空状态：引导去工作流编排并运行工作流。

## 风险

- 现有运行状态文本存在历史 mojibake 数据，前端仍需通过 `displayStatus` 归一化展示。
- 成本单价可能未配置，页面必须避免把 `$0.000000` 误导为真实成本。
- 当前没有实时推送，页面刷新才更新。
