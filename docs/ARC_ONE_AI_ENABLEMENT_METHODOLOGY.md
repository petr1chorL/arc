# ARC.ONE AI 赋能平台方法论沉淀

更新时间：2026-07-03

## 1. 方法论目的

本文沉淀本轮对话中形成的 ARC.ONE 产品与开发方法论，用于后续继续设计 AI 赋能平台时保持一致。

它回答三个问题：

1. AI 赋能平台到底在管理什么。
2. Agentic Workflow 应该如何设计、评分、审核和观测。
3. 后续开发如何在快落地和不误报能力之间取得平衡。

## 2. 第一性原理

AI 赋能不是“给每个岗位加一个聊天框”，而是把可重复的业务判断、信息处理和产出过程，转化为可编排、可评估、可追溯、可持续改进的工作系统。

因此平台的底层对象不是“Prompt”，而是：

- **业务场景**：一个有负责人、边界、输入、产出和衡量标准的问题。
- **工作流**：围绕业务成果组织起来的有向流程。
- **节点**：一次执行、判断、评估或人工决策。
- **Agent**：承担明确职责的版本化智能能力。
- **产出物**：节点产生的结构化、可追溯结果。
- **Rubric**：用于评价产出物质量的版本化量规。
- **Quality Gate**：决定产出物继续、重试、停止或进入人工审核的门禁。
- **Human Task**：需要系统用户明确通过或驳回的人工审核任务。
- **Trace**：一次运行的完整链路证据。

最小闭环是：

```text
业务目标
-> 工作流
-> 节点执行
-> 产出物
-> Rubric 评分
-> Quality Gate 路由
-> 人工审核或自动继续
-> 运行观测
-> 反馈沉淀为 Golden Sample
-> 迭代 Agent / Rubric / Workflow
```

## 3. 平台产品模型

ARC.ONE 应被理解为企业 Agentic Workflow 操作系统，而不是 Dify 的简单替代。

它包含以下核心模块：

| 模块 | 解决的问题 |
|---|---|
| Agent 资产 | 管理职责、模型、System Prompt、Tools、Skills、版本和停用状态 |
| 工作流编排 | 把 Agent、人工审核、质量门禁、工具、数据查询和完成节点编成流程 |
| 运行中心 | 展示每次 Workflow Run 的输入、状态、当前节点、耗时和结果 |
| 产出物 | 保存每个节点的结构化 Artifact，并能追溯来源 |
| 评估中心 | 管理 Rubric、Golden Sample、Evaluation 和回归测试 |
| 人工审核 | 让指定用户处理需要人工判断的任务 |
| 运行观测 | 用 Trace 还原运行链路、失败原因、节点状态和审计事件 |
| 模型 Provider | 管理模型接入配置和非密钥资产 |
| Tool / Skill | 管理 Agent 可调用能力 |
| Data Object | 管理节点之间交换的数据结构 |
| 通知运维 | 管理通知 Outbox、渠道和失败恢复 |
| 审计日志 | 记录关键操作事实 |

## 4. 工作流设计原则

### 4.1 工作流不是页面，而是业务闭环

工作流应该以业务成果为中心，而不是以技术节点为中心。

一个好工作流必须回答：

- 输入是什么。
- 每个节点承担什么职责。
- 每个节点产出什么。
- 谁或什么机制判断产出物是否合格。
- 不合格时如何重试、驳回、人工审核或终止。
- 最终产出物是什么。
- 如何复盘这次运行。

### 4.2 节点类型

建议保留以下节点类型：

- 手动触发。
- Agent。
- Tool 调用。
- 数据查询。
- 条件分支。
- 质量门禁。
- 人工审核。
- 代码执行。
- 等待节点。
- 流程完成。

### 4.3 编排体验

工作流模块应分为两个页面：

```text
工作流列表页
-> 选择某个工作流
-> 独立工作流编排页
```

列表页负责资产管理：

- 名称。
- 状态。
- 版本。
- 节点数。
- 连线数。
- 最近更新时间。
- 运行按钮。
- 编辑按钮。

编排页负责设计与发布：

- 当前工作流名称。
- 输入输出 Schema。
- 节点画布。
- 节点属性面板。
- 保存草稿。
- 发布版本。
- 运行工作流。
- 版本记录。

## 5. Agent 设计原则

一个 Agent 不应该只是一个 Prompt。每个 Agent 至少应有：

- 职责边界。
- 输入契约。
- 输出契约。
- System Prompt。
- 模型配置。
- Tools。
- Skills。
- 知识或上下文来源。
- 质量要求。
- 禁止事项。
- 失败处理方式。
- 发布版本。

Agent 草稿可以编辑；发布版本必须不可变。工作流运行应引用已发布版本，避免运行结果不可复现。

## 6. 评分体系设计

### 6.1 专业术语

这套体系建议使用以下术语：

- Rubric：评分量规。
- Dimension：评分维度。
- Weight：权重。
- Anchor：评分锚点。
- Threshold：通过阈值。
- Quality Gate：质量门禁。
- Golden Sample：黄金样本。
- Evaluation：评估记录。
- Regression Run：回归测试运行。

### 6.2 Rubric 模板

一个 Rubric 应包含：

```text
Rubric 名称
适用对象
版本
评分维度
每个维度权重
每个维度评分锚点
硬性门禁
自动通过阈值
人工复核阈值
失败原因分类
证据要求
样本集
```

### 6.3 权重依据

权重不能凭感觉拍脑袋，应来自四类依据：

1. **业务风险**：出错会带来多大损失。
2. **用户价值**：这个维度对最终使用者有多重要。
3. **可逆性**：错误是否容易发现和修正。
4. **历史数据**：过往失败、人工审核和 Golden Sample 中最常见的问题。

权重初版可以由专家给出，但必须通过运行数据持续校准。

### 6.4 评分锚点

不要只写“好/中/差”，要写可观察标准。

示例：

```text
5 分：结论完整，覆盖全部输入约束，引用证据充分，没有明显遗漏。
3 分：覆盖主要问题，但存在次要遗漏或证据不足。
1 分：结论不可用，遗漏关键约束，或与输入事实冲突。
```

### 6.5 自动评分与人工审核关系

自动评分负责规模化筛查，人工审核负责高风险判断。

建议路由：

```text
高分且无硬性问题 -> 自动通过
中间分或关键维度不确定 -> 人工审核
低分或硬性门禁失败 -> 驳回、重试或终止
```

## 7. 模板放在哪里

模板分三层：

### 7.1 Agent 模板

放在 Agent 资产中。

包括：

- System Prompt。
- 输入输出格式。
- 工具使用规范。
- 领域知识说明。
- 失败处理提示。

### 7.2 节点模板

放在工作流节点配置中。

包括：

- 节点输入字段。
- 节点输出字段。
- 节点运行参数。
- 人工审核说明。
- 节点级 Rubric 或 Quality Gate。

### 7.3 全链路规范

放在 Workflow Version 或 Workflow Contract 中。

包括：

- 工作流目标。
- 全局输入输出 Schema。
- 节点间数据对象。
- 全链路质量策略。
- 运行观测要求。
- 人工审核策略。
- 版本说明。

不建议做一个“万能总控 Agent”来包办全链路规范。更合理的是：

```text
Workflow Orchestrator 负责调度和状态
Agent 负责单点能力
Rubric / Gate 负责质量判断
Human Task 负责人工决策
Observability 负责事实复盘
```

## 8. 人工审核设计原则

当前版本应保持简单。

用户已经确认的方向：

- 人工审核节点中指定一个或多个系统用户。
- 对应用户登录后，在人工审核模块看到自己的待审队列。
- 不需要认领任务。
- 不需要转交任务。
- 不需要复杂会签。
- 只保留通过和驳回。
- 审核页重点展示上游一个或多个节点的产出物和详情。

人工审核页的核心信息：

- 当前任务。
- 所属工作流运行。
- 上游产出物列表。
- 产出物详情。
- 审核原因输入。
- 通过。
- 驳回。
- 审计记录。

## 9. 运行中心设计原则

运行工作流后，应自动进入运行中心，而不是停留在弹窗里。

运行中心应展示：

- 本次 Run ID。
- 所属工作流和版本。
- 当前状态。
- 当前节点。
- 工作流可视化链路。
- 节点状态颜色。
- 节点产出物。
- 错误原因。
- 排障建议。
- Trace ID。

节点状态建议：

```text
绿色：通过
黄色：等待或人工审核中
红色：失败
蓝色或灰色：运行中或未开始
```

## 10. V1.0 Lite 落地策略

为了快速落地，不追求大而全一次完成，应采用 V1.0 Lite。

V1.0 Lite 必须包含：

- 一个可登录 Workspace。
- 一组试点 Agent。
- 一个可运行 Workflow。
- 一个 Human Review 节点。
- 一个 Rubric。
- 一个 Golden Set。
- 一次 Workflow Run。
- 一次人工审核。
- 一次 Evaluation。
- 一条 Trace。
- 一份验收证据。

可以后置：

- 多租户复杂治理。
- 真实通知渠道。
- 大规模队列。
- 多模型路由。
- 高级权限矩阵。
- 自动优化 Agent。
- 企业级报表。

## 11. 开发方法论

### 11.1 先澄清，再实现

产品结构类问题先确认交互，不直接写代码。

推荐问题格式：

```text
在哪个页面？
用户点哪个按钮？
进入哪个页面或状态？
看到哪些信息？
哪些不要出现？
如何验收？
```

### 11.2 小步快跑

每轮只做一个端到端切片。

示例：

```text
工作流列表页
-> 点击编辑
-> 独立编排页
-> 测试
-> 浏览器验证
```

### 11.3 TDD 与验证

行为变化必须有测试。

缺陷处理流程：

```text
复现问题
找根因
写失败测试
实现最小修复
跑聚焦测试
跑 lint / build
浏览器验证
```

### 11.4 不误报能力

文档和回复不能把原型能力说成完整生产能力。

需要明确：

- 已实现。
- 可演示。
- 只是前端原型。
- 只是脚本能力。
- 需要真实模型密钥。
- 需要后续后端能力。

## 12. 对抗式审查清单

每个功能完成前问：

- 它是否制造了“看起来完成但实际不可用”的错觉。
- 是否绕过 Workspace 隔离。
- 是否绕过权限、审计或版本不可变。
- 是否泄露密钥、账号、Token 或私密配置。
- 是否只覆盖快乐路径。
- 是否让用户承担了本该自动化验证的成本。
- 是否把页面堆复杂了，而不是按用户任务拆页面。
- 是否阻碍未来接入 LangChain、LangGraph、DeepAgents 或真实模型运行。

## 13. 与 Dify、LangChain、LangGraph、DeepAgents 的关系

ARC.ONE 和 Dify 在“编排 Agent / Workflow”层面相似，但 ARC.ONE 更强调企业治理闭环：

- 资产版本。
- Rubric。
- Golden Sample。
- Human Review。
- Trace。
- Audit。
- Workspace 权限。
- 质量运营。

后续接入 LangChain、LangGraph、DeepAgents 是可行的，但它们应作为运行时或编排引擎能力，而不是替代 ARC.ONE 的产品模型。

建议边界：

```text
ARC.ONE：资产、治理、评估、审计、运行观测、用户体验
LangGraph：复杂状态图和 Agent 运行编排
LangChain：工具、模型、检索和基础链路组件
DeepAgents：深度任务执行或多 Agent 执行模式
```

## 14. 后续工作方式建议

为了提升会话效率：

- 长会话只做阶段总结，不继续承担高频开发。
- 新会话先读交接文档，再做任务。
- 用户每次尽量给一个明确问题。
- Agent 每次先给 3-5 条计划，再改文件。
- 复杂功能先拆 PRD / Issue。
- UI 调整以浏览器截图和用户标注为准。
- 每次完成必须说明：改了什么、怎么验证、还剩什么。
