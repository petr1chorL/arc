# V1.0 Lite 业务方用户手册

> 更新时间：2026-06-29
> 适用对象：业务负责人、产品负责人、审核人、试点操作人

## 你要完成什么

你要用 ARC.ONE 跑通一条 AI 赋能试点流程，并留下可验收证据：

```text
输入业务材料
-> 运行 Workflow
-> 查看 Agent 产出
-> 完成人工审核
-> 保存评分记录
-> 查看运行 Trace
-> 记录结论和问题
```

## 第一性原理

本手册不要求你理解底层代码，只要求你验证一件事：平台是否能把一个真实业务输入变成可审核、可评分、可追溯的产出物。

## 对抗式审查

- 只登录成功不算验收。
- 只看到页面不算验收。
- 没有 Run ID、Human Task ID、Evaluation ID、Trace ID，不算验收。
- 如果人工审核被跳过，不算验收。
- 如果产出物无法被业务方判断是否可用，不算验收。

## 1. 准备信息

开始前请确认你有：

- 试点账号。
- 试点 Workspace 名称。
- 本次试点业务材料。
- 审核人姓名。
- 管理员提供的 V1.0 Lite 地址，默认是 `http://127.0.0.1:4173`。

默认样例输入：

```json
{
  "sourceNotes": "安克 AI 课程笔记与个人思维导图摘要",
  "businessContext": "希望构建一个企业 AI 赋能平台，用于编排 Agent、人工审核和质量评分",
  "desiredOutput": "平台落地路线与一个可执行试点流程",
  "riskConcerns": "不要大而全失控，先快速试点；质量评分体系要可落地"
}
```

- AI 赋能工作流设计。

## 4. 启动 Workflow Run

1. 打开试点 Workflow。
2. 点击运行或触发入口。
3. 粘贴本次输入 JSON。
4. 启动运行。
5. 记录 Run ID。

运行后可能出现两种情况：

- 自动节点完成后进入人工审核。
- 某个节点失败，需要管理员排查。

## 5. 完成人工审核

1. 打开人工审核页面。
2. 找到本次 Run 对应的审核任务。
3. 点击认领任务。
4. 阅读产出物和上游信息。
5. 选择通过、修改后通过或驳回。
6. 填写审核理由。

审核时重点看：

- 业务目标是否清楚。
- Workflow 节点是否能真实执行。
- 高风险点是否进入人工审核。
- Rubric 是否能指导改进。
- 是否存在密钥、隐私或敏感信息泄露。

## 6. 运行或查看 Evaluation

1. 打开评估中心。
2. 找到本次产出物或手动发起评分。
3. 使用试点 Rubric 评分。
4. 确认评分结果保存。
5. 记录 Evaluation ID。

评分不通过时：

- 记录低分维度。
- 记录是否进入修复任务。
- 记录是否需要重新运行 Workflow。

## 7. 查看运行观测

1. 打开 Observability 或运行观测页面。
2. 搜索或进入本次 Run。
3. 检查节点状态、Trace、失败原因、Token 或成本信息。
4. 确认能从观测页回到产出物或审核任务。
5. 记录 Trace ID。

## 8. 查看通知或 Outbox

1. 打开通知运维页面。
2. 查看是否有与本次审核、SLA 或运行相关的通知记录。
3. 如果有失败通知，记录失败码和排障建议。

如果没有通知记录，也可以通过，但需要记录“当前无待处理通知”。

## 9. 判断本次是否通过

全部满足才算通过：

- 有 Run ID。
- 有 Human Task ID。
- 有 Evaluation ID。
- 有 Trace ID。
- 有业务方认可或明确驳回的产出物结论。
- 阻断问题已记录到 `docs/V1_LITE_PILOT_ISSUE_LOG.md`。
