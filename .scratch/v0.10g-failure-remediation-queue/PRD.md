# V0.10G 失败原因修复队列 PRD

## 问题陈述

V0.10F 已经能把最新 Regression Run 的失败样本聚类成原因摘要，但用户仍需要自己判断下一步该处理哪个原因、怎么处理、处理后如何复测。评估中心需要把失败原因摘要进一步转成可行动的修复队列。

## 解决方案

在 `Failure Pattern Summary` 下方展示 `Failure Remediation Queue`。每个失败原因组生成一个修复项，展示优先级、关联样本数、最低分、建议动作、代表样本 ID 和复测提示。

## 用户故事

- 作为质量负责人，我能从失败原因聚类直接看到优先处理顺序。
- 作为 Agent / Workflow 构建者，我能知道每个失败原因应该先补什么材料或规则。
- 作为验收人，我能用代表样本 ID 回到 Regression Run 详情中复核问题。

## 实施决策

- 本切片只做前端确定性队列，不新增后端表。
- 优先级由失败样本数和最低分计算：
  - `P0`：样本数 >= 3 或最低分 < 50。
  - `P1`：样本数 >= 2 或最低分 < 60。
  - `P2`：其他失败原因组。
- 修复项来源于 V0.10F 的 `FailurePatternCluster`，不重复计算失败样本。
- 队列最多展示 3 项，与失败原因组数量保持一致。

## 测试决策

- 扩展 `src/pages/Evaluations.test.tsx`，在已有失败聚类样本下断言修复队列展示。
- 测试必须先失败于缺少 `Failure Remediation Queue` 区块。
- 浏览器验收使用真实登录、真实 Regression Run 和真实评估中心页面。

## 范围外

- 不创建持久化 Issue / Task。
- 不接入飞书、邮件或外部通知。
- 不自动重跑 Regression Run。
- 不调用 LLM 生成修复建议。
